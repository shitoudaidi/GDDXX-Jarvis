# Jarvis Design Skill Stack v0.1

Date: 2026-06-20

## Goal

Build Jarvis as a cinematic product interface: a private AI presence, not a generic chatbot page. The first design target is the main cockpit screen: blue tactical HUD, central particle intelligence, restrained information panels, voice-reactive motion.

## GitHub Skills Reviewed

- `taste-skill`: broad anti-slop frontend guidance for Codex and Claude Code. Useful as a general design quality bar.
- `baoyu-design`: local agent skill for polished UI mockups, prototypes, decks, and wireframes as self-contained HTML.
- `designer-skills`: broad design skill collection covering research, systems, UI, interaction, and critique.
- `ui-design-brain`: component best practices for production UI, dashboards, forms, navigation, states, and accessibility.
- `motion-design-skill`: motion principles for timing, easing, choreography, interaction feedback, and ambient animation.
- `extract-design-system`: design token extraction from public websites, useful later if we choose reference sites.

## Local Skills We Will Use First

1. `industrial-brutalist-ui`
   Primary visual grammar for Jarvis: tactical telemetry, rigid grids, HUD compartments, terminal precision, technical framing, scanline and hardware-like details.

2. `high-end-visual-design`
   Material quality and motion polish: machined surfaces, strong spatial rhythm, refined transitions, no generic template feeling.

3. `motion-design` from LottieFiles
   Motion identity: premium, calm, deliberate. Use primary, secondary, and ambient motion layers. No random animation.

4. `image-to-code`
   Image-first workflow. For the main interface, generate visual references first, analyze them, then implement.

5. `ui-design-brain`
   Component behavior reference only: buttons, panels, empty/loading/error states, navigation, accessible controls. It should not control the Jarvis visual identity.

6. `impeccable` / `design-taste-frontend`
   Final anti-slop review: avoid generic AI gradients, card spam, fake complexity, low contrast, over-rounded cards, and meaningless micro-labels.

## Design Workflow

1. Write the visual direction.
2. Generate three main-screen design directions.
3. Pick one direction or combine the strongest parts.
4. Generate a cleaner hero/main-cockpit reference.
5. Analyze spacing, palette, panels, particle core, and motion states.
6. Build the first interactive prototype.
7. Verify with screenshots and motion checks.

## Jarvis-Specific Overrides

- The interface is product UI, but the first screen may feel cinematic.
- Blue and cyan are identity colors, not generic AI decoration.
- Red/orange is reserved for warning and emergency states only.
- Avoid copying Iron Man UI directly. Keep the feeling: private lab, holographic intelligence, always-on assistant.
- The central particle body is the hero. Side panels support it and must not dominate.

## Anti-Patterns

- Generic purple-blue AI gradient hero.
- Three equal feature cards.
- Full-screen fake dashboard clutter.
- Too many glowing lines without hierarchy.
- Game UI styling.
- Huge rounded glass cards everywhere.
- Decorative labels that do not carry real state or information.
- All-caps text everywhere.
- Static UI with no breathing, listening, thinking, or speaking states.
